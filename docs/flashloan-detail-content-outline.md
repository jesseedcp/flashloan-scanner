# 闪电贷识别详情页重构文档

## 1. 文档目标

这份文档不是讨论记录，而是给另一个 AI 或工程实现者直接执行的重构说明。

目标只有一个：

> 把当前交易详情页重构成一页面向“闪电贷识别取证”的详情页，而不是字段堆叠页、控制台下钻页或泛化 dashboard。

主参考站点：

- `BlockSec Phalcon Explorer`

次参考：

- 你之前提供的长报告图，用于借鉴“章节感”和“单列阅读路径”

不再作为主参考的：

- `Dune`
- `Vercel`
- 普通 dashboard / observability 控制台

原因：

- 当前页面本质上是“单笔链上交易的取证详情页”
- Phalcon 处理的是同类问题：这笔交易是什么、资产怎么变、资金怎么流、调用怎么走、底稿在哪里

---

## 2. 页面定位

新页面的定位是：

> 一页讲清楚这笔交易为什么被识别为一笔真实、完整、可解释的闪电贷。

用户看完以后，应当能回答五个问题：

1. 这笔交易被识别成了什么。
2. 系统为什么认为它成立。
3. 关键资产借了什么、还了什么。
4. 资产在关键参与方之间如何流动。
5. 如果继续追问，底层证据在哪里。

---

## 3. 当前页面的根本问题

### 3.1 不是视觉问题，而是内容问题

当前页面最根本的问题不是“丑”，而是内容组织方式错了。

当前页面更像：

- 有哪个数据结构，就给它一个模块
- 有哪些字段，就换一种形式再展示一遍

而不是：

- 我要证明什么
- 这块证据在证明什么
- 哪些内容应该在主阅读路径里
- 哪些内容应该下沉到底稿

### 3.2 具体问题

当前页面存在这些结构性问题：

1. 同一信息反复出现
   - 顶部摘要、关键依据、证据模块、底稿区在重复讲同一批事实

2. 模块没有明确回答的问题
   - 观众看到“调用路径证据”“规则验证证据”“资产记录证据”，但不清楚每块到底证明什么

3. 底层技术细节抢了主路径
   - selector、frame 数、原始字段出现过早

4. 主图不对
   - 之前的“关键地址关系图”不如 `Fund Flow` 图有解释力

---

## 4. 新的主信息架构

页面主路径严格改成五节，顺序固定如下：

1. `Basic Information` / `基本信息`
2. `Balance Changes` / `资产变化`
3. `Fund Flow` / `资产流向图`
4. `Invocation Flow` / `调用调用树`
5. `Raw Evidence` / `原始证据`

这是直接参考 `BlockSec Phalcon Explorer` 得出的结论。

这五节分别回答：

1. 这笔交易是什么，以及系统把它识别成了什么。
2. 哪些地址的哪些资产发生了变化。
3. 资产在关键参与方之间怎么流动。
4. 内部调用如何展开。
5. 如果继续追问，底层证据在哪里。

**注意：**

- 不再按 `summary / trace / rules / asset` 这种工程结构分节
- 不再保留“关键地址关系图”作为主图
- `Fund Flow` 应当成为页面主图之一

---

## 5. 三刀删减原则

这是当前页面去重和收主线最优先的三刀，执行时必须先做。

### 第一刀：删掉所有“这块回答 / 本块结论”

原因：

- 每个模块都先解释自己，会让页面非常啰嗦
- 观众会感觉每节都在自我介绍
- 这些解释卡并没有增加事实信息量

执行原则：

- 每节标题下面只保留一行导语
- 不再保留两张解释卡

### 第二刀：删掉“过程时间线”

原因：

- 主过程图应当是 `Invocation Flow`
- 时间线继续留在主路径，会和调用树重复回答“过程怎么发生”

执行原则：

- 主路径不再展示过程时间线
- 如果要保留，只能下沉到 `Raw Evidence`
- 如果实现代价高，直接删除，不必保留

### 第三刀：删掉“关键依据”里重复计数项

特别要删的：

- `涉及协议`
- `识别交互`
- 任何已经在 `Basic Information` 中清楚表达的计数

原因：

- 顶部已经给过这些事实
- 再讲一遍，只是重复

执行原则：

- `关键依据` 不再作为独立大节存在
- 结论摘要只保留真正的判定依据：
  - callback / 回调路径
  - repayment / 归还路径
  - settlement / 结算闭环
  - strict asset record / 严格资产记录

---

## 6. 第一节：Basic Information / 基本信息

### 6.1 这一节回答什么

> 这笔交易是什么，以及系统把它识别成了什么。

### 6.2 这一节应该长什么样

结构参考 `Phalcon` 的 `Basic Information` 表格式摘要。

页面表现建议：

- 页头显示交易哈希
- 下方是两列或三列的摘要表
- 一行一个事实，不用卡片拼盘

### 6.3 建议字段

第一层：识别信息

- `扫描结论`
- `识别类型`
  - `Flash Loan Interaction`
- `命中协议`
- `协议入口`
- `区块号`
- `时间`

第二层：执行信息

- `发起地址`
- `接收 / 回调地址`
- `区块内位置`
- `Gas Used`
- `Gas Price`
- `Transaction Fee`

可选：

- `Trace 状态`
- `内部调用帧数`

### 6.4 不建议放进主信息区的字段

- `Value`
- `Nonce`
- `Input Data`
- `Event Count`
- `Internal Transaction Count`

这些不是完全不能展示，而是不应该进入第一页主信息。

### 6.5 建议结论句

> 本笔交易识别为 Aave V3 闪电贷交互，入口为 `flashLoanSimple`，系统最终判定为严格通过。

---

## 7. 第二节：Balance Changes / 资产变化

### 7.1 这一节回答什么

> 哪些地址的哪些资产发生了变化。

### 7.2 这一节的定位

这是详情页里第一张主表。

它不是单纯的“资产列表”，而是：

> 借入、归还和费用变化的结构化表格。

### 7.3 建议表头

- 地址
- 地址角色
- 资产
- 资产记录类型
- 借入数量
- 归还数量
- premium / fee
- 变化方向
- 是否为严格资产记录

### 7.4 说明原则

- 不照搬 `USD value`
- 不照搬 `Profit / Cost / Revenue`
- 不做伪利润分析
- 这张表只服务于“借还闭环”说明

### 7.5 建议结论句

> 该交易的关键资产变化集中在 provider、回调接收者和归还目标之间，其中 1 条严格资产记录形成了借入与归还闭环。

---

## 8. 第三节：Fund Flow / 资产流向图

### 8.1 这一节回答什么

> 资产在关键参与方之间是怎么流动的。

### 8.2 这是页面主图之一

这一节必须被视为主图，不是附属图。

原因：

- 它比“关键地址关系图”更能解释闪电贷闭环
- 它直接回答：
  - 资产从哪里来
  - 经历哪些关键参与方
  - 最终回到哪里

### 8.3 设计原则

参考 `Phalcon` 的 `Fund Flow`，但不要照搬复杂度。

当前可实现目标应当是：

> 基于 `trace_summary.asset_flows`、关键地址角色和资产记录生成的“闪电贷识别版流向图”。

### 8.4 图中必须表达的内容

- provider / pool 节点
- receiver / callback 相关节点
- intermediary protocol / pair / pool 节点
- repayment target 节点
- borrow / swap / repay 这几类边
- asset symbol / amount / direction

### 8.5 旧图如何处理

旧的“关键地址关系图”不再作为主图。

处理方案：

- 直接删除
- 或降级到 `Raw Evidence` / 附属说明中

**不要**继续把它当主图使用。

### 8.6 建议结论句

> 资产流向图显示借入资产经过回调相关路径使用后，最终沿归还路径回到协议侧，支持该交易形成完整闪电贷闭环。

---

## 9. 第四节：Invocation Flow / 调用调用树

### 9.1 这一节回答什么

> 内部调用是如何展开的。

### 9.2 这一节的定位

这是调用取证主图，不是说明卡片区。

它要更像 `Phalcon` 的调用树，而不是之前那种多卡片散列式 trace 展示。

### 9.3 应该展示什么

- 调用层级
- 关键调用节点
- callback 路径
- repayment 路径
- token action
- 关键 event 节点

### 9.4 视觉原则

- 树形或层级展开
- 关键节点高亮
- 次级节点弱化
- 能一眼看出：
  - 协议入口
  - callback
  - repayment
  - settlement
  - error

### 9.5 建议结论句

> 调用树显示该交易从协议入口进入，在回调子树中执行资产操作，并最终沿归还路径完成结算，调用层面支持其为完整闪电贷交互。

---

## 10. 第五节：Raw Evidence / 原始证据

### 10.1 这一节回答什么

> 如果继续追问，底层原始证据在哪里。

### 10.2 这一节的原则

全部折叠，不进入主阅读路径。

### 10.3 保留内容

只保留这几类底稿：

1. 完整内部调用链原始展开
2. 原始协议交互字段
3. Input Data
4. 其他浏览器式原始字段

### 10.4 不要让底稿抢主线

以下内容都不应在主阅读路径直出：

- selector 级细节
- frame 数
- 原始 verification notes 大段文字
- Value / Nonce / Input Data 等低相关字段

---

## 11. 术语规范

可以保留但要解释的术语：

- trace
- callback
- repayment
- strict

应该主文案人话化的词：

- interaction -> 协议交互
- asset leg -> 资产记录
- strict leg -> 严格资产记录
- settlement -> 结算闭环
- exclusion reason -> 排除原因
- debt opening -> 开债迹象

---

## 12. 参考图借鉴说明

### 12.1 以 Phalcon 为主参考

原因：

- 它是交易取证详情页
- 和当前闪电贷详情页属于同一问题类型
- 它按“交易是什么 -> 资产怎么变 -> 资产怎么流 -> 调用怎么走 -> 底稿在哪里”组织页面

### 12.2 可以直接借的结构

- `Basic Information`
- `Balance Changes`
- `Fund Flow`
- `Invocation Flow`

### 12.3 不该直接照搬的部分

- `Profit / Cost / Revenue`
- `Builder`
- `State Changes`
- `Gas Profiler`
- 其他泛浏览器字段作为主路径内容

原因：

- 它们不属于“闪电贷识别成立”的核心证据链
- 会把页面带成浏览器详情页，而不是取证报告页

---

## 13. 给另一个 AI 的执行优先级

如果另一个 AI 要直接执行，建议顺序如下：

### 第一步：先删

1. 删掉所有“这块回答 / 本块结论”
2. 删掉“过程时间线”主路径
3. 删掉“关键依据”里重复计数项
4. 删掉旧的“关键地址关系图”主图地位

### 第二步：重排主路径

把详情页重排成：

1. Basic Information
2. Balance Changes
3. Fund Flow
4. Invocation Flow
5. Raw Evidence

### 第三步：实现策略

- `Basic Information`：用表格，不用卡片拼盘
- `Balance Changes`：用主表，不做利润分析
- `Fund Flow`：作为主图，优先于地址关系图
- `Invocation Flow`：做树，不做卡片列表
- `Raw Evidence`：全部折叠

### 第四步：再做视觉

等信息架构稳定后，再考虑：

- 标题样式
- 色彩系统
- 悬浮提示
- 动效

不要在结构未定时先修视觉。

---

## 14. 一句话总结

新的闪电贷详情页不应该是：

> 我有哪些字段，所以我展示哪些字段。

而应该是：

> 我已经判定这是一笔真实的闪电贷，所以我要把这个判定如何成立讲清楚。

---

## 15. 页面线框草图（给实现者）

以下不是视觉稿，而是信息架构线框。

```text
[ 页头 ]
交易哈希 / 复制 / 返回 / 协议徽标

[ 01 Basic Information ]
表格式摘要
- 扫描结论 | 识别类型 | 命中协议 | 协议入口
- 区块号 | 时间 | 发起地址 | 接收/回调地址
- 区块内位置 | Gas Used | Gas Price | Transaction Fee

[ 02 Balance Changes ]
一张主表
- 地址 | 角色 | 资产 | 借入 | 归还 | fee/premium | strict
下方一行短结论

[ 03 Fund Flow ]
一张主图
- 从 provider / pool 出发
- 经过 receiver / callback / intermediary
- 回到 repayment target / protocol
下方一行短结论

[ 04 Invocation Flow ]
一张调用树主图
- 协议入口高亮
- callback 子树高亮
- repayment 路径高亮
- event / token action 显示为次级信息
下方一行短结论

[ 05 Raw Evidence ]
折叠区
- 原始协议交互
- 完整内部调用链
- Input Data
- 其他浏览器式原始字段
```

阅读顺序必须是单列向下，不要再回到双栏卡片拼盘。

## 16. 组件层实现约束

### 16.1 Basic Information

实现形式：

- 采用表格或 definition list 风格
- 不使用统计卡拼盘
- 每一行一个事实
- 地址支持复制与 hover 完整展示

不允许：

- 顶部再放一排重复的“关键依据”卡片
- 再放一张单独的大结论卡

### 16.2 Balance Changes

实现形式：

- 使用主表格
- 地址列优先显示角色，再显示短地址
- 数值列统一右对齐
- strict 用轻量标签，不用大胶囊

建议表内增加：

- `方向` 列
  - borrow / repay / fee / net

不允许：

- 再把同一条资产记录拆成许多卡片
- 表和下方再重复一遍同样的统计

### 16.3 Fund Flow

实现形式：

- SVG 主图
- 横向流向图优先
- 边必须有方向箭头
- 节点必须有角色名，不只显示地址

图中分层：

- 主节点：provider / receiver / callback / repayment target
- 次节点：intermediary pool / pair / protocol
- 边标签：asset symbol + amount + action

实现目标：

- 一眼看出借入
- 一眼看出中间路径
- 一眼看出归还闭环

不允许：

- 退回成“谁和谁有关”的静态关系图
- 只画节点，不画资产方向

### 16.4 Invocation Flow

实现形式：

- 树形调用图
- 默认展示关键层级
- 允许展开更多

节点级别：

- 一级：协议入口
- 二级：关键子调用
- 三级：callback / repayment / token actions
- event 节点作为辅助信息显示

高亮规则：

- 协议入口
- callback 子树
- repayment 路径
- settlement 相关节点
- error 节点

不允许：

- 用一大列卡片假装调用树
- 让 selector 或 frame 数比调用关系更抢眼

### 16.5 Raw Evidence

实现形式：

- 全部折叠
- 默认关闭
- 一次只展开一个分组也可以

内容优先级：

1. 原始协议交互
2. 完整内部调用链
3. Input Data
4. 其他技术字段

不允许：

- Raw Evidence 默认展开
- 原始字段进入主阅读路径

## 17. 文案与命名约束

标题统一使用交易取证语境，不要再回到工程字段语境。

推荐标题：

- `Basic Information` / `基本信息`
- `Balance Changes` / `资产变化`
- `Fund Flow` / `资产流向图`
- `Invocation Flow` / `调用调用树`
- `Raw Evidence` / `原始证据`

推荐子文案写法：

- 基本信息：`用于确认这笔交易的识别结果、协议入口与执行上下文。`
- 资产变化：`用于查看关键地址在各资产上的借入、归还与费用变化。`
- 资产流向图：`用于说明借入资产如何在关键参与方之间流动，并最终形成归还闭环。`
- 调用调用树：`用于查看内部调用如何从协议入口展开到回调与归还路径。`
- 原始证据：`用于在追问时回看原始协议字段、完整调用链与浏览器式底稿。`

不推荐继续使用的标题：

- `协议识别证据`
- `调用路径证据`
- `规则验证证据`
- `资产记录证据`
- `关键参与地址`
- `关键地址关系图`

原因：

- 这些名字更像系统内部模块
- 不是交易取证页面的自然章节名

---

## 18. 给另一个 AI 的最小交付标准

如果另一个 AI 只能做一轮，最低要达到这 6 条：

1. 页面主路径改成单列
2. 首屏改成 `Basic Information` 表格式摘要
3. 主表改成 `Balance Changes`
4. 主图改成 `Fund Flow`
5. 调用图改成真正的树或层级图，而不是卡片列表
6. 原始字段全部折叠到 `Raw Evidence`

只有满足这 6 条，才算真正脱离当前旧详情页思路。
