# 前端演示控制台产品方案

> 文档角色：产品方案文档  
> 目标读者：负责继续实现的 AI / 工程同学  
> 文档用途：明确这个前端页面要做什么、为什么做、用户怎么用、第一版范围到哪里为止

## 1. 产品目标

本页面的目标不是做一个静态结果展示页，而是做一个适合课程项目演示的“实时扫描控制台”。

用户进入页面后，应当能够：

- 选择扫描参数
- 一次启动三协议扫描任务
- 在扫描过程中实时看到进度
- 在发现目标交易时立即看到结果输出
- 在扫描完成后查看明细结果

这个页面的核心价值是“可演示”。

也就是说，页面不仅要展示最终结果，还要展示“系统正在工作”的过程。相比静态 dashboard，这种形式更容易向老师或评审解释：

- 用户输入了什么
- 系统做了什么
- 系统发现了什么
- 最终产出了什么

## 2. 目标用户

第一版只考虑一类用户：

- 课程项目演示者

这个用户的核心诉求不是长期运维，而是：

- 快速配置一次扫描
- 稳定完成一次演示
- 清楚看到三协议同时扫描和实时发现结果

因此，第一版产品不需要考虑：

- 多用户协作
- 登录权限
- 历史任务管理
- 长时间后台运行

## 3. 核心使用场景

### 场景 1：课堂展示

用户在演示时打开页面，输入已经准备好的主网区块区间，点击开始扫描。页面随后同时展示：

- Aave V3 扫描进度
- Balancer V2 扫描进度
- Uniswap V2 扫描进度
- 实时发现的 candidate / verified / strict 结果

最终页面展示三协议都命中了真实样本。

### 场景 2：项目答辩

用户通过该页面展示系统不仅有静态结果，而且能实时运行。评审能够看到：

- 扫描任务确实被启动
- 每个协议都在推进
- 扫描过程中不断有结果输出
- 最终可以点进某笔交易查看详情

### 场景 3：快速复验

用户需要重新扫描一个给定区块窗口，快速确认系统是否还能识别出预期样本，而不需要进入数据库或命令行。

## 4. 产品定位

这是一个“扫描控制台”，不是“完整区块链浏览器”，也不是“长期运行的监控平台”。

第一版定位：

- 一个交互式演示页面
- 支持手动发起扫描
- 支持实时输出结果
- 支持查看交易详情

不做的事情：

- 不做链上全量浏览
- 不做复杂账户系统
- 不做大规模任务调度平台
- 不做运维后台

## 5. 产品成功标准

如果第一版完成，用户应该能够完成下面这条链路：

1. 打开页面
2. 填写扫描参数
3. 点击“开始扫描”
4. 实时看到三协议并发推进
5. 实时看到发现结果不断追加
6. 扫描完成后看到汇总结果
7. 点击任意一条结果查看详情

如果这条链路顺畅，就说明这个产品对课程项目演示是成功的。

## 6. 第一版功能范围

## 6.1 必做功能

### 功能 A：扫描参数配置

用户可以在页面中设置：

- 链 ID
- 起始区块
- 结束区块
- 是否启用 trace
- 要扫描的协议

第一版中，协议默认三选全开：

- `Aave V3`
- `Balancer V2`
- `Uniswap V2`

### 功能 B：启动扫描任务

用户点击开始按钮后，前端向后端发起一次扫描任务。

该任务是一个总任务，内部同时跑三个协议子任务。

### 功能 C：实时进度展示

页面应当实时显示：

- 总任务状态
- 三协议各自状态
- 当前处理区块
- 已发现多少 candidate
- 已发现多少 verified
- 已发现多少 strict

### 功能 D：实时结果输出

当系统识别到一条结果时，页面需要立即追加显示，而不是等全部扫描结束后一次性出现。

每条结果至少显示：

- `tx hash`
- `protocol`
- `block number`
- `candidate`
- `verified`
- `strict`

### 功能 E：结果详情查看

用户点击某条结果后，可以查看：

- 交易基本信息
- interaction 信息
- asset legs
- 验证状态
- verification notes

### 功能 F：扫描完成态

扫描结束后，页面应明确显示：

- 任务已完成
- 三协议各自结果统计
- 最终总结果统计

## 6.2 暂不做功能

第一版不做以下能力：

- 用户登录
- 多任务并发管理
- 历史任务列表
- 暂停 / 恢复 / 取消任务
- 断线自动恢复
- 权限系统
- 复杂搜索系统

## 7. 用户界面结构

建议第一版只做两个页面：

- `Scan Console`
- `Transaction Detail`

### 7.1 Scan Console

这是主页面，也是演示时最重要的页面。

建议包含 5 个区域：

#### 区域 1：扫描配置面板

内容包括：

- 链选择
- 起始区块输入框
- 结束区块输入框
- trace 开关
- 协议选择
- 开始扫描按钮

#### 区域 2：总任务概览

内容包括：

- 当前任务状态
- 总 candidate 数
- 总 verified 数
- 总 strict 数
- 已完成协议数

#### 区域 3：三协议进度卡片

分别展示：

- `Aave V3`
- `Balancer V2`
- `Uniswap V2`

每张卡片显示：

- 当前状态
- 当前扫描到的区块
- 当前已发现结果数
- 最近一条结果摘要

#### 区域 4：实时结果表

这是页面最核心的演示区域。

要求：

- 新结果到达时立即插入表格顶部
- 不需要刷新页面
- 行点击后进入详情

#### 区域 5：实时日志区

日志区不是必须复杂，但应能让用户感知“系统正在运行”。

例如：

- 某协议开始扫描
- 扫描推进到某个区块
- 发现一条 verified 结果
- 某协议扫描结束

### 7.2 Transaction Detail

详情页或右侧抽屉都可以，第一版优先考虑实现成本最低的方式。

详情页需要展示：

- `tx hash`
- `protocol`
- `block number`
- interaction 列表
- leg 列表
- candidate / verified / strict 状态
- 验证说明

## 8. 交互流程

推荐交互流程如下：

1. 用户打开控制台
2. 页面默认填入一组可演示的区块区间
3. 用户点击 `Start Scan`
4. 页面进入 running 状态
5. 三协议卡片开始分别更新进度
6. 实时结果表开始追加新结果
7. 全部协议完成后，总状态变为 completed
8. 用户点击任意结果查看详情

## 9. 实时通信方案

前后端通过 `WebSocket` 通信。

选择 WebSocket 的原因：

- 需要实时推送进度
- 需要实时推送发现结果
- 更适合做“控制台式”演示页面
- 相比轮询，交互更及时，观感更好

第一版交互方式：

- 前端建立 WebSocket 连接
- 用户点击开始扫描后，通过 WebSocket 发送任务参数
- 后端持续推送：
  - 任务开始
  - 协议进度
  - 新发现结果
  - 协议完成
  - 总任务完成

## 10. 后端产品要求

后端不是这个文档的重点，但为了让另一个 AI 能顺利实现，需要明确后端必须提供什么能力。

### 必要能力 1：支持一次任务同时扫三个协议

产品要求是“一次点击，三协议并发扫描”，而不是让用户分别点三次。

### 必要能力 2：支持实时推送

后端需要在扫描过程中持续向前端推送：

- 总任务状态
- 协议子任务状态
- 新发现结果
- 结束状态

### 必要能力 3：支持结果详情查询

扫描后，前端需要能够根据 `tx hash` 查询更完整的数据明细。

### 必要能力 4：与现有 scanner 复用

新页面不应要求重写扫描核心逻辑。后端应尽量复用现有：

- scanner
- verifier
- database 层
- report / result 查询能力

## 11. 推荐消息类型

为了便于另一个 AI 实现，建议 WebSocket 消息至少定义如下类型：

前端发送：

- `start_scan`

后端发送：

- `job_started`
- `job_progress`
- `protocol_progress`
- `finding`
- `protocol_completed`
- `job_completed`
- `job_failed`

其中最关键的是 `finding`，因为这是用户最关心的实时结果输出。

## 12. 默认演示参数

为了让页面一打开就能演示，建议前端默认填入已经验证过的真实主网区间：

- `chain_id = 1`
- `start_block = 22485844`
- `end_block = 22486844`
- protocols = `Aave V3 + Balancer V2 + Uniswap V2`

选择这段区间的原因是：

- 范围较小
- 已确认包含三协议真实样本
- 已被现有系统验证可命中

## 13. 产品优先级

### P0：必须完成

- 控制台页面
- 三协议并发扫描
- WebSocket 实时进度
- WebSocket 实时结果输出
- 结果详情查看

### P1：有时间可以补

- 更漂亮的图表
- 导出 JSON / CSV
- 历史结果列表
- 更好的错误提示

### P2：本阶段不做

- 用户系统
- 多任务调度平台
- 复杂权限
- 长时间后台任务管理

## 14. 交付给实现方的明确要求

如果把这份文档交给另一个 AI，它应当按下面原则执行：

- 不要把产品做成静态 dashboard
- 不要把三协议拆成三次独立启动
- 不要只在扫描结束后一次性展示结果
- 必须实现“开始扫描后实时输出”
- 必须突出三协议并发运行的演示效果
- 第一版优先保证稳定和可演示，不追求复杂平台能力

## 15. 最终一句话定义

第一版产品应被实现为：

**一个基于 WebSocket 的实时扫描演示控制台，支持用户一次启动三协议扫描，并在页面中实时看到进度、发现结果与交易详情。**
