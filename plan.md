# COMP5566 项目计划

## 项目题目

Understand Flash Loan and Its Applications in DeFi Ecosystem

## 1. 项目目标

构建一个可复现的以太坊闪电贷交易识别与分析流程，完成以下三件事：

- 解释闪电贷提供方与用户在单笔交易中的交互过程
- 设计并实现一个基于规则的闪电贷交易识别器
- 基于识别结果分析闪电贷在 DeFi 生态中的常见用途，并完成一个小型演示

## 2. 建议范围

为了保证项目能在截止前完成，建议严格控制范围：

- 公链范围：只做 Ethereum mainnet
- 协议范围：只做 `Aave V3`、`Balancer V2`、`Uniswap V2 flash swap`
- 方法范围：规则识别 + trace 验证，不做机器学习
- 数据范围：先跑一个已确认包含三协议样本的真实主网窗口，跑通后再决定是否扩展
- 成果范围：扫描器原型、前端演示页、结构化结果导出、图表分析、案例研究、报告、答辩 slides

## 3. 核心问题

项目需要回答下面几个问题：

1. 闪电贷在链上是如何完成借出、执行、归还的？
2. 如何从链上数据中识别闪电贷交易？
3. 收集到的闪电贷交易主要涉及哪些协议、代币和用途？
4. 规则型检测方法有哪些局限性？

## 4. 最终交付物

小组最终应提交：

- 闪电贷识别与分析源码
- 清洗后的数据集，格式可以是 CSV、SQLite 或 DuckDB
- 至少 4 到 6 张核心统计图表
- 2 到 3 个代表性案例分析
- 一个可复现的小型 demo
- 最终报告
- 10 分钟展示 slides
- Turnitin 查重结果，重复率低于 5%

## 5. 技术路线

### 5.1 整体流程

建议按照下面的 3 个阶段实现：

1. 候选交易收集
   - 从已知闪电贷协议的合约、事件或关键调用入手，先收集候选交易
   - 保存基础字段：交易哈希、区块号、时间戳、协议、代币、金额等

2. 交易验证
   - 对候选交易做 trace 验证
   - 确认交易中是否出现以下模式：
     - 资金从 lender 转出给 borrower
     - borrower 在回调中执行自定义逻辑
     - 同一笔交易内完成归还，或满足协议要求的结算条件

3. 结果分析
   - 对识别出的交易按协议、代币、金额、是否成功、可能用途进行分类
   - 生成图表与案例分析

### 5.2 第一版检测规则

第一版必须用可解释的规则，不要一开始做复杂模型。

- Aave：
  - 检测与 flash loan 相关的调用
  - 区分 `candidate`、`verified`、`strict`
  - 对 `interestRateModes != 0` 的情况单独标记，避免直接混入严格闪电贷统计

- Balancer：
  - 检测 Vault 的 flash loan 行为
  - 验证官方 Vault 事件、回调和归还路径是否成立

- Uniswap V2：
  - 检测带回调逻辑的 flash swap 模式
  - 验证 `swap(data != empty)`、回调路径和不变量恢复语义

### 5.3 需要保存的数据字段

当前实现不再只以单张 transaction 表为核心，而是按 `transaction / interaction / asset leg` 三层保存数据。

建议最少保存以下字段：

- `tx_hash`
- `block_number`
- `timestamp`
- `protocol`
- `entrypoint`
- `provider_address`
- `receiver_address`
- `callback_target`
- `candidate_level`
- `success`
- `verified`
- `strict`
- `asset_address`
- `amount_borrowed`
- `amount_repaid`
- `premium_amount`
- `fee_amount`
- `settlement_mode`
- `exclusion_reason`
- `verification_notes`

### 5.4 分析输出

至少产出以下分析结果：

- 按协议统计的闪电贷交易数量
- 按代币统计的分布情况
- 金额分布
- 时间趋势图
- 成功与失败交易比例
- 2 到 3 个典型交易案例的逐步解释

## 11. 风险与应对

### 风险 1：范围过大

应对：

- 只做以太坊主网
- 只做 3 个主流协议
- 先从短时间窗口开始

### 风险 2：数据采集过慢或结果不完整

应对：

- 先从协议相关候选交易入手，不做全链无差别扫描
- 先用小样本验证流程，再扩大范围

### 风险 3：误报过多

应对：

- 增加 trace 验证
- 对结果做人工抽样核查
- 在报告中明确 precision 的测量方式

### 风险 4：demo 复杂度过高

应对：

- demo 以现有扫描控制台和详情页为主，不再扩展成复杂套利或本地 fork 系统
- 优先保证 demo 可讲清楚、可复现、能支撑答辩叙事
